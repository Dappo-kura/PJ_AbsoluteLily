# 仕様書: 都市伝説JK百合ホラーゲーム (α版)

## 1. 概要
- **ジャンル**: QTEサイコホラーノベル
- **プラットフォーム**: Windows (Godot Engine)
- **ターゲット**: 百合ホラー好き、QTEアクション好き
- **コンセプト**: 「恐怖」と「絆」の天秤。選択肢とQTEで運命が変わるマルチエンディングノベル。

## 2. ゲームループ
1.  **ノベルパート**:
    - 会話、ナレーションの読み進め。
    - 選択肢によるパラメータ変動（絆、穢れ、恐怖など）とフラグ管理。
    - **恐怖値 (Fear)** が100を超えるとゲームオーバー。
2.  **QTEパート (怪異との対峙)**:
    - 収縮するリングがターゲットゾーンに入った瞬間にボタンを押す (参考：孵道)。
    - 成功：恐怖値減少、物語進行。
    - 失敗：恐怖値上昇、バッドルートへの分岐フラグ。
3.  **エンディング分岐**:
    - パラメータとフラグの状態でエンディングが決定。
    - TRUE, NORMAL, BAD (各怪異堕ち), HAREM (隠し) など。

## 3. システム仕様

### 3.1 パラメータ管理
| 変数名 | 日本語名 | 初期値 | 説明 |
| :--- | :--- | :--- | :--- |
| `fear` | 恐怖度 | 0 | 100以上で即ゲームオーバー。QTE失敗や危険な選択で上昇。 |
| `kizuna` | 絆 | 0 | ヒロインとの親密度。TRUE END到達に必要。 |
| `kegare` | 穢れ | 0 | 怪異への同調度。怪異堕ちENDに関わる。 |
| `san` | 正気度? | - | (現状のモックでは `kyoko` (狂行/恐慌?) が使われている模様。`fear`と統合するか検討) |
| `flags` | フラグ | {} | 各怪異（口裂け、テケテケ等）への好感度やイベント進行フラグ。 |

### 3.2 ノベルシステム
- **テキスト表示**: 一文字ずつ表示、スキップ機能、バックログ。
- **選択肢**: 2〜3択。制限時間付きの選択肢も実装可能にする。
- **立ち絵**: 通常、恐怖、笑顔などの差分切り替え。
- **背景**: シーンごとの背景画像。
- **演出**:
    - 画面振動 (シェイク)
    - ノイズ/グリッチエフェクト
    - 赤色オーバーレイ (ダメージ/恐怖)
    - 暗転フェード

### 3.3 QTEシステム (Shrinking Ring)
- **JSONによる詳細パラメータ指定**:
    - `duration`: 収縮にかかる時間 (ms)
    - `size`: リングの初期サイズ (px)
    - `zone_ratio`: 成功判定ゾーンの広さ (0.0〜1.0)
    - `text`: 中央に表示するテキスト (例: "逃げろ!", "耐えろ!")
    - `random_offset`: 出現位置のランダム幅
- **表示**: 画面中央などにリングが表示され、徐々に収縮する。
- **判定**:
    - `Success`: 指定のターゲットゾーン内で入力。
    - `Fail`: ゾーン外で入力、または時間切れ。
- **難易度調整**: リングの縮小速度、ターゲットゾーンの広さで調整。
- **連撃**: 連続してQTEが発生する場合がある。

### 3.4 セーブ＆ロード
- **スロット**: 複数セーブスロット。
- **オートセーブ**: 章の区切りなどで自動セーブ。
- **システムデータ**: エンディング回想、CGギャラリーのアンロック状況を保存。

### 3.5 フローチャート
- 既読済みのシーンをフローチャートで確認可能。
- 任意の既読シーンから再開可能（「ここから再開」機能）。

### 3.6 演出コマンドシステム (Scenario Direction System)
シナリオの「演出」をJSONデータ内で完結・制御できるようにする。
- **基本構造**: テキスト表示だけでなく、コマンドを配列で処理する。
- **コマンド例**:
    - `fade_in`, `fade_out`: 画面フェード
    - `shake`: 画面振動 (強さ、時間指定)
    - `play_se`, `play_bgm`: 音声再生 (フェードイン/アウト、音量指定)
    - `wait`: 待ち時間 (演出のタイミング合わせ)
    - `char_move`: 立ち絵の移動、ズーム、透明度変更
    - `glitch`: グリッチエフェクトの適用
    - `flash`: フラッシュバック（赤、白）

## 4. 画面構成

### 4.1 画面遷移図 (Screen Flow)
```mermaid
graph TD
    Boot((起動)) --> Title[タイトル画面]
    
    Title -->|New Game| Init[初期化/プロローグ]
    Title -->|Load| LoadMenu[ロード画面]
    Title -->|Settings| Settings[設定画面]
    Title -->|Gallery| Gallery[ギャラリー]
    Title -->|Exit| Quit((終了))

    Init --> Game[ゲーム本編\n(ノベルパート)]
    LoadMenu -->|Select Data| Game
    
    subgraph GameLoop [ゲームループ]
        Game -->|Choice| Branch{選択肢}
        Branch --> Game
        
        Game -->|Event| QTE[QTEパート\n(怪異対峙)]
        QTE -->|Success| Game
        QTE -->|Fail| Game
        
        Game -->|Menu Open| InGameMenu[インゲームメニュー]
        InGameMenu -->|Save| SaveMenu[セーブ画面]
        InGameMenu -->|Load| LoadMenu
        InGameMenu -->|Config| Settings
        InGameMenu -->|Return| Game
        InGameMenu -->|To Title| Title
    end

    Game -->|Fear >= 100| GameOver[GAME OVER]
    Game -->|Scenario End| Ending[エンディング]
    
    GameOver --> Title
    Ending --> Title
```

### 4.2 各画面詳細
1.  **タイトル画面**:
    - 「ニューゲーム」「ロード」「設定」「ギャラリー」「終了」
    - クリア状況に応じて背景変化。
2.  **ゲーム画面**:
    - 背景レイヤー、キャラクターレイヤー、UIレイヤー（メッセージウィンドウ）。
    - 画面端にパラメータ表示（オプションで非表示可）。
3.  **設定画面**:
    - 音量設定 (BGM, SE, Voice)
    - テキスト速度
    - 画面モード (フルスクリーン/ウィンドウ)
4.  **メニュー画面 (インゲーム)**:
    - セーブ/ロード、バックログ、設定、タイトルへ戻る。

## 5. アセット要件
- **画像**:
    - 立ち絵 (麻衣、百合香、各怪異) + 表情差分
    - 背景 (学校、商店街、踏切、神社、など) + 時間帯差分/異界差分
    - イベントCG (重要なシーンの一枚絵)
    - UI素材 (メッセージウィンドウ、ボタン、QTEリング)
- **音声**:
    - BGM (日常、不穏、恐怖、感動)
    - SE (足音、悲鳴、心臓音、インターホン、など)

## 6. 技術スタック
- **エンジン**: Godot Engine 4.x
- **言語**: GDScript
- **データ形式**:
    - シナリオデータ: JSON (演出コマンド対応の拡張フォーマット)
    - セーブデータ: user:// (ConfigFie または JSON)

## 7. 開発・プレビュー環境 (重要)
シナリオ演出の品質を高めるため、以下の機能を実装する。
- **ホットリロード**: ゲームを再起動せずに、JSONファイルの変更を即座に反映して現在のシーンをリロードする機能。
- **デバッグUI**:
    - シーンジャンプ機能
    - 早送り / 巻き戻し (ステート復元)
    - 演出コマンドのテスト実行
    - **QTE調整モード**:
        - QTEパラメータ（速度、サイズ、テキスト）をGUIでその場で変更し、テストプレイする機能。
        - 調整した値をログに出力し、JSONへの反映を容易にする。

## 8. 想定されるリスクと対策
承認済みリスクアセスメント（2025/01/10時点）
1.  **シナリオデータ構造の変更**: 実装途中での構造変更は全データの修正コスト大。初期にJSONフォーマットを確定する。
2.  **アセット命名規則の不統一**: リンク切れ防止のため、`bg_loc_time` 等の規則を徹底する。
3.  **QTEの根幹変更**: 全く別のアクションへの変更は作り直しになるため避ける。パラメータ調整で対応する。

## 9. シナリオ管理の拡張方針 (将来の展望)
将来的にシナリオが数万ワード（文庫本1〜2冊分以上）の規模になることを見据え、以下の管理手法への移行を検討する。

### 9.1 ファイル分割 (File Splitting)
- **チャプター単位の管理**: `scenario.json` を1つの巨大なファイルにするのではなく、`scenario_ch1.json`, `scenario_ch2.json` のように章単位で分割する。
- **動的読み込み**: `ScenarioManager` が必要になったタイミングで該当する章のファイルを読み込み、メモリ消費を抑える。

### 9.2 執筆環境のマスター化 (Master Data Management)
- **スプレッドシート管理**: JSON を直接編集するのではなく、Google スプレッドシートや Excel で執筆・校正を行う。
- **自動変換スクリプト**: スプレッドシートのデータをプロジェクト用 JSON に一括変換する Python スクリプト等を導入し、量産体制を整える。

### 9.3 テキストとロジックの分離 (Localization & Scaling)
- **テキストキーの導入**: JSON 内には演出ロジックと「テキストID」のみを記述し、実際のセリフは CSV 形式（Godot の Localization 機能）で管理する。
- **多言語対応の基盤化**: セリフを CSV に逃がすことで、将来的な英語・他言語対応を容易にする。
- **整合性チェック**: ID とセリフの紐付けを自動チェックする仕組みを導入し、数万ワード規模でのリンク切れを防ぐ。
