# 操作感・入力レスポンス改善（2026-07-20 / 最大2サイクル）

## サイクル1: ADV（ノベルパート）入力レスポンス＆待機フィードバックの一貫化

### 背景
- テーマ「入力受付・レスポンス・コヨーテタイム・入力バッファ等の操作感」。成功条件は「見た目・反応・細部まで一貫し、完成度の高さを感じられること」。
- ディレクター分析とCodex独立レビューが独立に一致した課題: ①ADV連打の二重送りによる行スキップ、②待機インジケータがアニメ無しで存在感が薄い、③マウス↔キーボードの操作範囲不一致（選択肢のフォーカス欠如）。
- 変更規模「小」。プレイ中最も接触頻度が高いADVの手触りから着手し、QTEは次サイクルへ回すことで各サイクルを1〜2ファイルに収める。

### 採用（サイクル1・対象: `godot_project/scripts/game/main.gd` のみ）
1. **テキスト送りの二重入力ガード（入力バッファ/デバウンス）**
   - `complete_text_display()` 実行直後の一定時間（約150ms）、`advance_text()` を抑止する。
   - ラピッド2連クリック／ダブルクリックで「全文表示→即次送り」となり表示直後の行を読み飛ばす問題を解消（Codex中・#2）。
   - `Time.get_ticks_msec()` ベースのガード時刻で判定。スキップ(Ctrl)経路は対象外（意図的な高速送りのため）。
2. **待機インジケータのアニメーション化（視覚フィードバックの一貫性・完成度）**
   - `click_indicator` を表示時に点滅/浮遊するループTweenでアニメーションさせ、非表示時に停止・リセット。
   - 「次に進める」ことが常に一貫した反応で伝わるようにする。
3. **通常選択肢のキーボード導線（レスポンスの一貫性）**
   - `show_choices()` で先頭ボタンへ `grab_focus()`。上下キー/Enterでの選択を成立させ、マウスとキーボードの操作範囲を揃える。

### 不採用・次サイクル送り
- **QTEのキーボード導線・フォーカス／コヨーテタイム(grace window)・円形ヒットテスト・hover/押下フィードバック** — Codex Top3の中心だが、`qte_controller.gd`＋`qte_choice.gd`にまたがり、単一確定ガードや世代IDの配慮が要る。**サイクル2で集中実装**。
- **AキーのAUTO状態ガード漏れ（Codex中・#5）／SKIP解除後のAUTO待機リセット（低・#9）** — 軽微な追加修正。サイクル1のmain.gd変更に付随して低リスクで同梱できるため**採用に格上げ**（下記実装指示に含める）。
- **フォーカス喪失時のQTEタイマー停止（中・#8）／InputMap統一（低・#10）** — 効果は中だが範囲が広がるため保留（人間確認項目として記録）。

### 対象ファイル（想定1件）
- `godot_project/scripts/game/main.gd`

## サイクル2: QTE操作感の一貫化（キーボード導線・円形ヒット・状態フィードバック・コヨーテタイム）

### 背景
- サイクル1でADV側の手触りを整えたので、看板メカニクスのQTEを揃えて全体の完成度・一貫性を上げる。
- Codex独立レビュー（サイクル2）とディレクター分析が一致した高/中インパクト課題から、**安全に小規模で実装できるもの**を選定。

### 採用（対象: `qte_choice.gd` / `qte_choice.tscn` / `qte_controller.gd` の3ファイル）
1. **円形ヒットテスト**（`qte_choice`）: 見た目の円（半径64px相当）内だけをクリック可能にする。矩形四隅の透明領域での誤選択（約49.7%が円外）を解消。→ 細部の完成度。
2. **状態フィードバックの統一**（`qte_choice`）: hover=外周ハイライト/明度上昇、focus=フォーカスリング（キーボード可視化）、pressed/selected=短いフラッシュ or 縮小。無効化は再描画を伴う`set_interactable()`でリング凍結を防止。→ 反応の一貫性。
3. **キーボード導線**（`qte_choice`+`qte_controller`）: 選択肢をフォーカス可能にし、Enter/Spaceで選択。QTE開始時に**最初に出現した選択肢へ`grab_focus`**（キーボード操作を実用化）。上下左右の移動は既定のfocus neighborに委ねる。
4. **コヨーテタイム（grace window）**（`qte_controller`）: `GRACE_TIME≈0.12s`。締切(`timer<=0`)後もこの猶予中は選択を有効とし、`timer<=-GRACE_TIME`で初めて失敗確定。低FPS/フレーム順序による締切直前入力の取りこぼしを防止。タイマーバーは0でクランプ済み。
5. **リング進捗の一貫化**（`qte_choice`+`qte_controller`）: 各リングの割合を「出現時残り」ではなく「全体残り/全体duration」基準に統一（`setup`へ remaining と total を渡す）。遅出選択肢が満円から始まる不整合を解消。
6. **世代ID(run_id)による安全ガード**（`qte_controller`）: `start_qte`毎に`run_id`を更新し、非同期スポーンと`finish_qte`の各await後で世代照合。grace追加・再開始時の旧coroutine混入/二重emitを防止（Codex最重要リスク対策・単一確定を維持）。

### 不採用・保留（今サイクル）
- **完全な状態機械(enum 4状態)への全面移行** — 効果はあるが範囲拡大。今回はrun_id＋既存`is_active`＋grace判定で単一確定を担保し、必要最小限に留める。
- **Alt+Tabフォーカス喪失でのタイマー/動画停止**（Codex中・#9） — 効果中だがNOTIFICATION処理の追加検証が必要。人間確認/次回課題として記録。
- **InputMap整理（`qte_action`重複解消）**（低・#10） — Button単一経路を維持する方針と整合。別途小変更に切り出し保留。
- **初期フォーカスの当て先ポリシー** — Codex推奨に沿い「最初に出現した選択肢」へフォーカスするが、スポーン順シャッフルで死亡選択肢に乗りEnter即死し得る。フォーカスリングで可視化した上で、是非を**人間確認項目**とする。

### 対象ファイル（想定3件）
- `godot_project/scripts/game/qte_choice.gd`
- `godot_project/scenes/game/qte_choice.tscn`
- `godot_project/scripts/game/qte_controller.gd`

## （旧テーマ）サイクル1: QTEの公平性バグ修正＋残り時間の視覚化

### 背景
- Codex独立レビューは実行失敗（終了コード2・出力なし）。Claude単独分析で方針決定。
- 本作は全7章・マルチエンド・QTEルート分岐＝周回前提。QTEは看板メカニクス（参考「孵り道」）。

### 採用: QTE改善（`qte_controller.gd` / `scenes/game/qte.tscn`）
**理由**
- `qte_controller.gd` の選択肢は0.2〜0.5秒間隔で順次出現（最大11個＝最大約5.5秒）、一方 `duration` は最低4秒保証。
  → **正解選択肢が遅れて出現するとタイムアウトが先に来て理不尽死**する明確なバグ。最優先。
- 残り時間の全体UIが無く、時間プレッシャーが視覚化されていない（面白さ・操作性に直撃）。
- 単一システムで完結し「中規模」に収まる。

**変更内容**
1. 出現スケジュールを先に確定し、`duration = 全出現所要時間 + 反応猶予(最低4秒)` に変更。正解が最後でも必ず反応可能に。
2. 選択肢総数の上限を `randi_range(3,6)`（最大8個）に抑制。グリッド過密を回避。
3. 各選択肢の残り時間リングを共有デッドラインに整合（出現時の残り時間を渡す）。
4. 画面上部に全体カウントダウンバーを追加。残り少で赤へ変化。

### 不採用（今サイクル）
- **ADV基本操作（スキップ/オート/バックログ/テキスト速度）** — 影響大だが複数ファイル＋セーブ挙動に波及。サイクル2候補。
- **ステータスのゲージバー化** — 影響中。サイクル3候補。
- **音量設定UI** — 影響小。保留。

### 対象ファイル（想定2件）
- `godot_project/scripts/game/qte_controller.gd`
- `godot_project/scenes/game/qte.tscn`

## サイクル2: ADVオート/スキップ送りシステム

### 背景
- Codex独立レビューは3回連続で実行失敗（終了コード2・出力なし）。Claude単独分析で方針決定。
- 本作はマルチエンド＋QTE分岐で周回前提。既読再読のテンポが最大の課題（参考「死因」のADV操作感）。

### 採用: ADVオート/スキップ（`main.gd` 中心）
**理由**
- テキスト送りが全画面クリックのみで、オート/スキップ/速度調整が皆無。周回テンポへの改善効果が最大。
- 単一システムで完結し「中規模」に収まる。in-session設定のためセーブ形式に影響しない。

**変更内容**
1. **オートモード**: キー切替（A）。テキスト表示完了後、一定待機（約1.2秒）で自動的に次へ進む。
2. **スキップモード**: Ctrl押下中は高速で送る（表示中は瞬時全文表示→短間隔で次へ）。
3. **ガード**: メニュー/選択肢/QTE/ゲームオーバー/エンディング表示中はオート・スキップを無効化（選択肢の自動選択はしない）。
4. **画面表示**: 「AUTO」「SKIP」インジケータを画面に表示（コード構築）。

### 不採用（今サイクル）
- **ステータスのゲージバー化** — サイクル3候補。
- **設定UI・音量UI** — サイクル3候補（時間があれば）。

### 対象ファイル（想定1件）
- `godot_project/scripts/game/main.gd`
（インジケータもコード構築のためtscn変更なしを想定。入力はInputでキー直接判定し `project.godot` は変更しない方針）

# 新テーマ実行: UI・UX改善（メニュー・HUD・フィードバック表示）

## サイクル1: ゲーム内HUDの視認性・フィードバック改善

### 背景
- Codex独立レビューは環境エラー(spawn EPERM)で失敗 → 別サブエージェントで独立レビューを代行。
- ディレクター分析と独立レビューが**独立に一致**した課題群（HUDレイアウト破綻・fear視認性・操作ガイド不在）は、
  いずれも「ゲーム内HUD」という単一システムに収まり、本テーマ「HUD・フィードバック表示の分かりやすさ」の中心。
- 過去サイクルでADVオート/スキップ・QTEタイマーバーは実装済み。今回はそこを土台にHUDの見せ方を底上げする。

### 採用: HUD視認性＆フィードバック強化（`main.tscn` + `main.gd` の HUD 1システム）
**理由**
- fear≥100でゲームオーバーの最重要指標が、レイアウト破綻＋極小フォント＋数値のみで危険域が伝わらない。
  ゲームオーバーの納得感（＝ホラーADVの手触り）に直結。
- 操作ガイド不在はオート/スキップ等の既存改善機能を「発見不能」にしており、費用対効果が高い。
- 単一システム（HUD）で完結し「中規模」に収まる。セーブ形式・依存・アセットに触れない。

**変更内容**
1. **Statusレイアウト是正**: `main.tscn` StatusLeft/Right の Image 巨大オフセット(1710 / -1771等)を修正し、
   立ち絵アイコンをコンテナ内へ。フォントサイズを14→可読サイズへ引き上げ。
2. **fear/kizunaのゲージバー化**: 数値ラベルに加え ProgressBar 等でバー表示。fearが高域(例:70/90)で色を
   黄→赤へ変化させ危険度を可視化（fear≥100=ゲームオーバーの予兆フィードバック）。パラメータ変化時に軽い
   演出（数値の一瞬の強調 or バーのトゥイーン）。
3. **操作ヒント常設**: 現状空の TopBar（`main.tscn:114-120`）に「クリック:送り / A:オート / Ctrl:スキップ /
   Esc:メニュー」の操作ガイドを常設（または半透明の小ヘルプ）。既存AUTO/SKIPインジケータと整合。

### 不採用（今サイクル → サイクル2候補）
- **QTE選択肢の色以外の判別手がかり追加** — レビュアー最優先だが`qte_choice`描画ロジック寄りで本テーマ
  （メニュー・HUD・フィードバック）の中心から外れる。UI/UX×QTEとして次サイクル最有力候補。
- **セーブ完了トースト＋上書き確認ダイアログ** — フィードバック系で本テーマ適合。次サイクル候補。
- **ゲームオーバー時のリトライ/コンティニュー** — 体験改善大だがセーブ・進行制御に波及。保留。
- **選択肢ボタンのスタイル＋暗幕** — 次サイクル候補。

### 対象ファイル（想定2件）
- `godot_project/scenes/game/main.tscn`（HUDレイアウト・ゲージノード追加）
- `godot_project/scripts/game/main.gd`（`update_parameter_display` のゲージ/色更新、TopBarヒント）

# 総合ブラッシュアップ再実行（2026-07-19）

## サイクル1: セッション内QTEリトライシステム

### 背景

- 既存作業ツリーにはQTE公平性、ADVオート/スキップ、HUD視認性の未コミット改善があり、これらは保全する。
- 独立レビューは、QTE失敗がバッドエンドへ直結する一方で手動セーブしかなく、再挑戦時の進行ロスが大きいと指摘した。
- 視聴覚素材不足は重大だが、新規背景・怪異・音源を一度に整備すると「1機能・1システム単位」の変更規模を超えやすいため、後続サイクル候補とする。

### 採用: セーブ形式を変えないQTE直前チェックポイント＋リトライ導線

**理由**

- 失敗ルートを見る緊張感は残しつつ、ゲームオーバー/バッドエンド後の反復作業を減らし、QTEへ再挑戦しやすくする。
- 既存の `ScenarioManager.start_scene_at()` とゲーム状態管理を再利用でき、依存追加・外部通信・永続セーブ形式変更なしで完結する。
- 操作性、テンポ、面白さに同時に効き、本作の看板システムであるQTEを1システム単位で改善できる。

**変更内容**

1. QTE開始直前に、現在シーン/イベント位置とゲーム進行値をディープコピーしたセッション内チェックポイントを作る。
2. QTE成功/ハーレム時はチェックポイントを破棄し、失敗時だけ保持する。
3. 失敗によるゲームオーバーまたはバッドエンド画面に「QTE直前からやり直す」ボタンを表示する。
4. リトライ時はパラメータ、フラグ、隠し値、QTE成功数、シナリオ位置を復元し、終了オーバーレイと終了状態を解除して同じQTEを再開する。
5. チェックポイントはメモリ内のみとし、既存セーブスロットや `user://` のファイル形式・ファイル数を変更しない。

### 採用時の安全条件

- `Dictionary.duplicate(true)` 等でネスト状態を分離し、失敗ルート中の変更がチェックポイントを汚染しないこと。
- リトライ復元中に `parameter_changed` / `game_over` の副作用が誤発火しない順序を設計すること。
- 通常エンディング、QTE成功後、タイトル帰還ではリトライ導線を表示しないこと。
- 既存の未コミット4ファイルを破棄・巻き戻ししないこと。

### 不採用（今サイクル）

- **背景・怪異画像の一括追加** — プレイヤー価値は最大級だが、素材制作と全章への割当を含めると今回の中規模変更を越えやすい。後続サイクルでロケーション群を絞って実施候補。
- **BGM/SE最小実装** — 音源が不足し、新規素材が必要。QTEリトライの検証後に演出システムとして独立実施候補。
- **QTE色当て構造の再設計** — 読解型QTEとして重要だが、既存QTE公平性差分と同時変更すると評価軸が混ざるため次候補。
- **章番号・変換スキップ本文の修正** — 正本シナリオと変換規則の精査が必要で、進行不整合リスクがあるため別サイクル候補。

### 対象ファイル（想定3〜5件）

- `godot_project/scripts/autoload/game_manager.gd`
- `godot_project/scripts/autoload/scenario_manager.gd`
- `godot_project/scripts/game/main.gd`
- 必要な場合のみ、関連するテストスクリプト/シーン

変更見込みは100ファイル以内。依存追加、セーブ形式変更、アセット削除、外部通信は行わない。

## サイクル2: 主要ロケーション背景セット＋正本割当

### 採用

第2章以降で背景が帰り道のまま固定される問題を、主要4ロケーションの新規背景と変換ツールのキーワード割当で解消する。

**背景セット（Codex制作）**

1. 異界化したゲームセンター／古いプリクラ機
2. 暗い書店／奥書庫
3. 閉鎖モールのマネキン回廊／地下区画
4. 迷界駅ホーム／停車する帰還列車

既存背景に合わせた彩度の高いアニメ背景美術、16:9横長、人物なし、文字・ロゴ・透かしなしで統一する。

### 実装方針

- 画像生成スキルのbuilt-inモードでCodexが4素材を制作し、`godot_project/assets/images/` に保存する。
- `tools/scenario_converter.py` の `BG_KEYWORDS` に、具体語を汎用語より先に置いた割当を追加する。
- 生成物 `godot_project/data/scenario.json` は変換ツール経由で再生成し、直接編集しない。
- QTEルートの空タイトルシーンは直前背景を保持し、通常場面の場所転換時にだけ背景を更新する。
- C7のモール前広場・エンディングは既存の帰り道/エンディング画像を活用し、今回の4系統に含めない。

### 採用理由

- ホラーADVの場所理解と怪異のペイオフを最も直接改善する。
- 4素材＋変換キーワードという単一の背景システムに収まり、100ファイル上限以内。
- 正本変換経路を維持するため、次回シナリオ再生成でも割当が失われない。

### 不採用（今サイクル）

- **怪異立ち絵4体の同時追加** — 背景と同時に行うと制作・評価軸が広がるため次候補。
- **BGM/SE追加** — 音源制作とイベント設計を別システムとして扱う。
- **バックログ** — 操作性への効果は高いが、Criticalな背景問題を優先。
- **選択肢フォーカス/スタイル** — 小規模で有効だが、背景導入後の視認性を見て次サイクル候補。

- `tools/scenario_converter.py`
